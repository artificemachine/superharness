# Installing Superharness (for AI Agents)

You are an AI agent (Claude Code, Codex CLI, or similar) installing superharness
in the current project. Follow this document step by step.

Superharness is a cross-agent coordination protocol. It lets multiple AI agents
share contracts, handoffs, and failure memory on the same codebase. After install,
you and other agents will read `.superharness/contract.yaml` before starting work
and write handoffs when you stop.

---

## Step 1 — Install superharness

```bash
pipx install superharness
```

If `pipx` is not available:
```bash
# macOS
brew install pipx && pipx install superharness

# Linux
python3 -m pip install --user pipx && pipx install superharness

# Windows (PowerShell)
python -m pip install --user pipx
python -m pipx install superharness
# Add pipx bin dir to PATH if not already present:
python -m pipx ensurepath
```

If already installed, upgrade to latest:
```bash
pipx upgrade superharness
```

Verify:
```bash
superharness --version
```

## Step 2 — Ensure PATH

`pipx` installs entry points to `~/.local/bin`. If `superharness` is not found:
```bash
pipx ensurepath
# then open a new shell
```

## Step 3 — Detect the environment

Before asking the user anything, gather what you can observe:

### 3a. Project signals (scan the project directory)

| File/Dir present          | Means              |
|---------------------------|---------------------|
| `package.json`            | Node.js / TypeScript |
| `pyproject.toml`, `requirements.txt`, `setup.py` | Python |
| `Gemfile`                 | Ruby                |
| `go.mod`                  | Go                  |
| `Cargo.toml`              | Rust                |
| `pom.xml`, `build.gradle` | Java / Kotlin       |
| `docker-compose.yml`, `Dockerfile` | Containerized |
| `Makefile`, `Justfile`    | Build system        |
| `.terraform/`, `*.tf`     | Infrastructure-as-code |
| `serverless.yml`          | Serverless          |

Combine all detected stacks into a slash-separated string, e.g. `Python/Docker/Terraform`.

### 3b. Agent availability

```bash
command -v claude && echo "claude-code available"
command -v codex && echo "codex-cli available"
```

Set `primary_agent` to whichever is available. If both are available, set
`primary_agent` to yourself (the agent running this install).

### 3c. Git signals

```bash
git remote get-url origin 2>/dev/null
```

- Contains `github.com` → `repo: github`
- Contains `gitlab` → `repo: gitlab`
- Contains `bitbucket` → `repo: bitbucket`
- Otherwise → `repo: other`

Check for existing CI:
- `.github/workflows/` → `ci: github-actions`
- `.gitlab-ci.yml` → `ci: gitlab-ci`
- `Jenkinsfile` → `ci: jenkins`

### 3d. Team size

```bash
git log --format='%ae' --since='90 days ago' | sort -u | wc -l
```

- 1 → `team_size: solo`
- 2-5 → `team_size: small`
- 6+ → `team_size: team`

### 3e. Existing harness artifacts

Check for:
- `CLAUDE.md` → user already configures Claude Code
- `AGENTS.md` → user already configures Codex
- `.cursor/rules` → uses Cursor
- `.github/copilot-instructions.md` → uses Copilot
- `.superharness/` → already initialized (abort — tell user it exists)

### 3f. Project status

- Has releases/tags → `status: active`
- Has commits but no tags → `status: active`
- Empty or just initialized → `status: greenfield`
- README mentions "maintenance" or "deprecated" → `status: maintenance`

## Step 4 — Ask the user (only what you can't detect)

Ask these questions. Skip any where you already have a confident answer.

### Required — always ask:

**Autonomy level:**
> How much control do you want over agent actions?
> - **autonomous** — agents act without asking (fastest, less oversight)
> - **supervised** — agents explain what they'll do, then proceed unless you stop them
> - **approval-gated** — agents propose a plan and wait for your explicit approval before every task

This maps directly to superharness confirmation gates.

**Project goal:**
> What are you working on right now? (one sentence)
> This becomes the first contract goal. Example: "Migrate the API from Express to Fastify"

### Conditional — only ask if detection failed:

- **Tech stack** — only if no recognizable project files were found
- **Primary agent** — only if neither `claude` nor `codex` is in PATH
- **Watcher** — only on macOS: "Want a background watcher that auto-dispatches tasks? (yes/no)"

### Never ask:

- Role titles, personality, preferences
- Anything the filesystem already answered
- Anything that doesn't change superharness behavior

## Step 5 — Write profile.yaml

Write profile.yaml to a **temp file at the project root** — `.superharness/` does
not exist yet (init creates it in Step 6). Pass the temp file to `--from-profile`
and init will copy it into `.superharness/profile.yaml` automatically.

```yaml
# Generated by superharness agent-install
# Edit manually or re-run: superharness init --detect
project_name: "My Project"    # detected or inferred from project files
created: YYYY-MM-DD
autonomy: supervised          # autonomous | supervised | approval-gated
primary_agent: claude-code    # claude-code | codex-cli
stack: Python/Docker          # detected from project files
repo: github                  # detected from git remote
ci: github-actions            # detected from CI config files
team_size: solo               # solo | small | team
existing_harness:             # list of pre-existing agent configs found
  - claude-md
  - agents-md
```

Write this to a temp file at the project root, e.g. `/tmp/superharness-profile.yaml`
or `./superharness-profile.yaml`. Do NOT write to `.superharness/` — it doesn't exist yet.

## Step 6 — Run superharness init

Use `--from-profile` to pass the profile.yaml you wrote in Step 5. Init creates
`.superharness/` and copies the profile in automatically:

```bash
cd /path/to/user/project
superharness init --from-profile /tmp/superharness-profile.yaml
```

If `CLAUDE.md` or `AGENTS.md` already exist, `init` will skip them (good —
don't overwrite the user's existing config).

Alternatively, use `--detect` to have superharness auto-detect the stack and
status from the project files (skips the profile):

```bash
superharness init --detect
```

## Step 7 — Tailor CLAUDE.md and AGENTS.md

After init creates the base files, append project-specific context you observed:

### For CLAUDE.md, add:

- Test command (e.g. `pytest tests/`, `npm test`, `cargo test`)
- Build command if applicable
- Linting command if a linter config exists
- Key directories and their purpose (only the important ones, max 5)
- Any conventions you noticed (e.g. "uses conventional commits", "monorepo with packages/")

### For AGENTS.md, add the same information.

Keep both files under 80 lines. Don't duplicate what the agent will discover
by reading the code. Focus on things that save the agent time on first contact.

## Step 8 — Create the first contract

Using the project goal from Step 4:

```bash
superharness task create \
  --project . \
  --id first-task \
  --title "USER'S GOAL HERE" \
  --owner PRIMARY_AGENT
```

## Step 9 — Wire autonomy level

Based on the user's autonomy choice, set environment variables. Tell the user
to add these to their shell profile:

**autonomous:**
```bash
export SUPERHARNESS_CONFIRM_NON_INTERACTIVE=YES
export SUPERHARNESS_CONFIRM_SKIP_PERMISSIONS=YES
```

**supervised:**
```bash
export SUPERHARNESS_CONFIRM_NON_INTERACTIVE=YES
# No skip-permissions — agent explains before acting
```

**approval-gated:**
```bash
# No env vars — all confirmation gates active (default)
```

## Step 10 — Verify and report

```bash
superharness doctor --project .
```

Tell the user:
- What was detected (stack, repo, team size)
- What was configured (autonomy level, primary agent)
- What they can do next (`superharness contract today`, `superharness delegate`)
- Where to find the profile (`cat .superharness/profile.yaml`)

---

## If the user already has superharness initialized

If `.superharness/` exists but `profile.yaml` does not, run only Steps 3-5
to generate the profile retroactively. Do not re-run init.

## If the user wants to change their profile

```bash
# Edit directly:
$EDITOR .superharness/profile.yaml

# Or re-detect from project files:
superharness init --detect
```

## Platform notes

- **macOS**: Offer launchd watcher setup (`superharness watcher-worker --project .`)
- **Linux**: Mention systemd unit file in `scripts/superharness-watcher@.service`
- **Docker/CI**: Use `--non-interactive` flag with explicit env var confirmation
- **Windows (native)**: superharness runs natively on Windows — no WSL or bash required.

### Windows native install (copy-paste ready)

```powershell
# 1. Install pipx (if not present)
python -m pip install --user pipx
python -m pipx ensurepath
# Open a new PowerShell window, then:

# 2. Install superharness
pipx install superharness

# 3. Verify
superharness --version

# 4. Bootstrap project
cd C:\path\to\your\project
superharness init
superharness doctor

# 5. Start the watcher (background via Windows Task Scheduler)
superharness watcher-worker --project .
# Or run in the foreground:
superharness watch --foreground --project .
```

### Windows watcher background service

`superharness watcher-worker --project .` registers a Windows Task Scheduler task
that runs the inbox watcher on a repeating interval.  No bash, no WSL, no launchd.

To remove the scheduled task:
```powershell
schtasks /Delete /TN "SuperharnessWatcher_*" /F
```

### Windows notes for agents

- All core commands (`init`, `doctor`, `contract`, `task`, `delegate`, `enqueue`, `watch`,
  `discuss`, `recall`, `hygiene`, `verify`, `close`) run natively on Windows.
- The `delegate` command uses `subprocess.run` instead of `os.execvp` — exit codes are
  propagated correctly.
- Lock paths use `%TEMP%` (e.g. `C:\Users\<user>\AppData\Local\Temp`) rather than `/tmp`.
- `shux doctor` will show `winsvc` as the watcher backend on Windows.
- Agent CLI detection (`claude`, `codex`) searches both standard PATH and common Windows
  install locations (`%LOCALAPPDATA%\Programs\Python\Scripts`, `%APPDATA%\Python\Scripts`).
