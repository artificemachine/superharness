## Fixed this run (all on PR #56)

1. **Migration v33 FK guard was cosmetic** — `PRAGMA foreign_key_check` ran in a
   `finally` block after the transaction had already committed, so a detected
   violation could not roll back. A live install sat at v33 with 3 real FK
   violations. Fixed: check moved inside the transaction, scoped to rebuilt
   tables; orphans NULLed during rebuild; migration v34 repairs already-migrated
   DBs. Verified by replaying v32→v34 against a copy of a real 10 260-task
   install: 0 violations, zero rows lost.
2. **CI security scan was never running.** `shipguard==0.3.2` crashes at import
   on Python 3.11 (`SyntaxError`, PEP 701 f-string backslash — 3.12+ only), and
   `security.yml` pins 3.11. `| tee` under `bash -e` (no `pipefail`) swallowed the
   crash and reported green. Fixed both: `set -o pipefail` + pin bumped to 0.5.2.
   **CI has been reporting a passing security scan while performing none.**
3. **`CONTRIBUTING.md` setup was broken** — `pip install -e ".[dev]"` installs
   nothing (no `[project.optional-dependencies]`; dev deps are PEP 735
   `[dependency-groups]`), so the next step failed with pytest not found.
4. **Windows CI red since v1.80.0** — `test_transcript_tail.py` wrote fixtures in
   text mode, so Windows translated LF→CRLF and byte-offset assertions broke
   (108 != 105). Product was correct; the test assumed Unix line endings.

## Open blockers, ranked by interview impact

### 1. Browser → RCE chain in the dashboard (CRITICAL, verified end-to-end)
- `_expected_origin()` (`dashboard-ui.py:1702`) returns `f"http://{Host header}"` —
  the expected origin is derived from the attacker-controlled request, so it
  always matches. `:1714` skips the check entirely when Origin is absent.
- `GET /` is unauthenticated and injects the auth token into the page (`:1682`).
- `/api/discussion/*/close` (`:3165`) and `*/create-task` (`:3187`) call no
  `_verify_mutation_auth()` at all, yet `subprocess.run(["shux", ...])`.
- Loopback binding is correctly enforced but does not defend against a browser
  the attacker already controls (DNS rebinding).
- Conditional on the operator running `shux dashboard` and visiting a hostile
  page — but the dashboard is a headline feature.

### 2. `TOKEN_RE` accepts path traversal (CRITICAL)
`commands/task.py:28` — `^[A-Za-z0-9._/-]+$` accepts `../../../../tmp/pwn` and `..`.
Flows into worktree path construction (`inbox_dispatch.py:216-222`) and
`shutil.rmtree` on close (`close.py:256-271`) with no containment check.
A correct sanitizer already exists at `engine/worktree_ops.py:10-19` and is used
by `swarm.py` — it was simply never wired in here. Fixing this collapses several
other findings at once.

### 3. SECURITY.md documents a control that does not exist (FALSE claim)
`SECURITY.md:23`: *"Each bypass has its own confirmation gate."*
`delegate-to-claude.sh:35-38` appends `-p --dangerously-skip-permissions`
unconditionally on `--non-interactive`. `SUPERHARNESS_CONFIRM_SKIP_PERMISSIONS`
is never checked in the dispatch path — and `delegate.py:1430` does
`os.environ.setdefault(..., "YES")`, i.e. auto-grants it. `cli/delegate.sh`, the
file the doc names, does not exist. Documenting a control that isn't there is
worse than documenting none, because it stops the reader from checking.

### 4. `main` has zero branch protection (HIGH — operator action)
`protected: false`, `required_status_checks.contexts: []`, `rulesets: []`.
All 27 CI jobs are advisory. This is why a dead security scanner survived, and
why PR #55 merged with Windows red. The aggregator gates (`QA Gate`,
`Windows-Native Release Gate`) are already correctly written — they just aren't
required. Highest leverage-per-effort item in this report; it is a settings toggle.

### 5. Optimistic locking bypassed at 17 of 18 writers (HIGH)
`tasks_dao.py:260-273` implements correct `WHERE id = ? AND version = ?` locking.
It is the only guarded writer. `ConcurrencyError` is raised in one place and
**caught nowhere**. Watcher tick + dashboard write in the same window silently
produce a state neither actor chose. WAL serializes writes, not read-modify-write.

### 6. `shux status --fix` always destroys live discussions (CRITICAL, correctness)
`commands/status.py:516` declares `repaired = []`; there is **no `repaired.append`
anywhere in the file**. The guard at `:1152` is therefore always false and control
always falls through to force-close — immediately after printing that the agents
are alive.

### 7. Remaining
- `daemon.py` cluster: `watcher_pid` parsed at `:78`, never used → double watcher
  + permanent 5s respawn loop (`:66,78,99`); `_stop_daemon` orphans the watcher.
- `inbox_watch.py`: 4 661 lines, 86 top-level functions, 132 `except Exception`,
  0 classes. Two `conn.commit()` calls sit inside `except Exception: pass`.
- Schema has **1 CHECK constraint** across 27 tables; `tasks.status` is bare TEXT,
  so a typo'd status is accepted and the task becomes invisible-stuck.
- Telemetry declares 3 event types, emits 1.
- 78 remote branches; 1 non-semver tag (`adapter-payload-verified`).
- No screenshot above the fold; `airm2max` in 3 tracked files; GitHub license API
  still reports `NOASSERTION` >24h after the canonical Apache-2.0 landed.

## What this repo says about you (honest read)

The engineering instincts on display are strong and unusually well-documented —
the v33/v34 migration comments explain *why* the previous approach was wrong,
which is better than most production code. The SQLite source-of-truth claim is
defended by a real AST-based guard test. Telegram input handling is genuinely
well-defended. 1 154 of 1 196 commits are conventional.

The gap is between the core (excellent) and the edges (unaudited). Every
CRITICAL is on a surface that grew organically: the dashboard HTTP layer, the
daemon supervisor, a validation regex written once and reused everywhere. A
senior reviewer will read `db.py` and be impressed, then read
`dashboard-ui.py:1702` and conclude the security posture is aspirational rather
than designed.

The single most damaging item is not a CVE — it is that CI was reporting a green
security scan while running nothing, for an unknown number of releases. That is
the kind of finding that reframes every other green check in the repo.

Fix order: `TOKEN_RE` first (collapses several findings), then the dashboard auth
chain, then turn on branch protection, then the `daemon.py` cluster. The
architecture debt (locking, god module) is real but is a deliberate program of
work, not an interview blocker.
